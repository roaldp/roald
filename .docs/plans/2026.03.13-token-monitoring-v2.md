# Token Monitoring v2: Model Tracking, Process Visibility, Detailed Reports

## High Level Goal

Extend the existing token monitoring system to answer three questions the current version cannot:
1. **Which models** are being used per operation (and at what cost)?
2. **Which processes** ran when, for how long, and what did they cost (including timeline view)?
3. **Where exactly** is the money going -- per-invocation detail, not just aggregated totals?

The current system logs to `token_usage.jsonl` and aggregates by operation name, but all invocations of `full_pulse` collapse into one row. There is no model field, no session/run ID, and no way to see individual invocations or their timeline.

## Important Technical Notes

### Lean JSONL format

The JSONL format is kept intentionally lean for efficient storage and parsing. Each record contains only what's needed for cost attribution and timeline analysis:

```json
{"ts":"2026-03-13T17:21:03","op":"full_pulse","model":"sonnet","in":283,"out":2239,"cache_in":19358,"cache_read":264788,"cost":0.1865,"dur":67.26,"rid":"20260313T162000"}
```

Short field names to minimize disk/parse overhead at high write volumes. Fields:
- `ts` -- ISO timestamp (start time of invocation)
- `op` -- operation name (what process started this)
- `model` -- model used
- `in`, `out`, `cache_in`, `cache_read` -- token counts
- `cost` -- USD cost
- `dur` -- wall-clock seconds (duration)
- `rid` -- session identifier

`tool_starts` is dropped from the JSONL format -- it's useful for debugging but not for cost attribution or timeline analysis. Still logged in the EXEC END text log line.

This is a breaking change from the current verbose field names. The report script must handle both old (verbose) and new (lean) formats during transition.

### Backward compatibility of JSONL format

New fields (`model`, `run_id`) and renamed fields are added to new records. Old records are handled gracefully:
- `model` defaults to `"unknown"` when missing
- `rid` defaults to `"unknown"` when missing
- Normalization happens inside `load_usage_log`, **before** the since-filter. Each dict is normalized to lean keys first, then `ts` is used for timestamp filtering. All downstream code uses only lean keys.
- Field mapping: `timestamp`->`ts`, `operation`->`op`, `input_tokens`->`in`, `output_tokens`->`out`, `cache_creation_input_tokens`->`cache_in`, `cache_read_input_tokens`->`cache_read`, `cost_usd`->`cost`, `elapsed_s`->`dur`
- `aggregate_by_operation` updated to use lean keys: `entry.get("op")`, `entry.get("in")`, `entry.get("out")`, `entry.get("cache_in")`, `entry.get("cache_read")`, `entry.get("cost")`. Note: `print_report` takes `OperationSummary` objects so needs no key changes.

### Timeline timezone assumption

Timestamps in JSONL are local time without timezone info (from `datetime.now()`). Timeline bucketing assumes all records are in the same local timezone. This is fine for a single-machine daemon.

### Model column width

The `model` column in detail/model reports is truncated to 20 characters. If `claude_model` config contains a full model ID (e.g. `claude-sonnet-4-20250514`), it gets truncated in display. The full value is always in the JSONL.

### RUN_ID generation

`RUN_ID` is set once at module level using `datetime.now().strftime("%Y%m%dT%H%M%S")`. Written to JSONL as `rid`. Groups all invocations from one pulse.py session together. Not a UUID -- human-readable and sortable. In tests, mock `pulse.RUN_ID` or assert `rid` is a non-empty string matching `r"^\d{8}T\d{6}$"`.

### Flag interactions

`--detail`, `--by-model`, and `--timeline` are mutually exclusive (use `argparse.add_mutually_exclusive_group`). The default (no flag) keeps the existing operation-aggregated report. `--limit N` only applies to `--detail` (default 50, shows most recent N invocations).

### Timestamp parsing in reports

All timestamp parsing uses `datetime.fromisoformat()`. Entries with unparseable timestamps are skipped (consistent with existing `load_usage_log` behavior).
