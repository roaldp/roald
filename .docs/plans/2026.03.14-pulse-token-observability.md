# Plan: Pulse Token Observability

## High-Level Goal

Add granular token/context observability to `pulse.py` so we can see exactly where the ~400K tokens per pulse are spent. Current logging only captures aggregate totals. We need per-turn, per-tool breakdowns stored in log files — never injected into LLM prompts. Collect 2-3 days of data to inform the optimization plan at `.docs/plans/2026.03.13-reduce-pulse-token-consumption.md`.

### Key Discovery (from Phase 0 of the optimization plan)

A minimal `claude -p "say hello"` invocation uses only **~16K tokens** for system prompt + all 48 tool schemas + CLAUDE.md. The full pulse uses **~394K tokens**. This means:
- **System/tool overhead: ~16K (4%)** — unauthenticated MCP servers do NOT contribute tool schemas (their tools are not loaded)
- **Conversation content: ~378K (96%)** — accumulated across assistant messages, tool call inputs, and tool results
- `--allowedTools` does NOT filter which tools are loaded into context

The real cost drivers are the tool results (Gmail search results, email bodies, Slack messages, file reads) accumulating in the multi-turn conversation context.

### What the stream-json already provides

Each event from `claude --output-format stream-json --verbose` contains:

| Event type | Available data |
|---|---|
| `system` (subtype `init`) | `tools[]` (all tool names), `mcp_servers[]` (name + status), `model` |
| `assistant` | `message.usage` with `input_tokens`, `output_tokens`, `cache_creation_input_tokens`, `cache_read_input_tokens` per turn |
| `user` (tool_result) | `content` (the actual tool result text — can measure size) |
| `result` | `total_cost_usd`, `duration_ms`, `num_turns`, aggregate `usage`, `modelUsage` per model |

**Sub-agent behavior in stream-json:**
- When Claude uses the `Agent` tool, the sub-agent's events are **interleaved inline** in the parent's stream
- Sub-agent `system` events have `model: None` (vs parent's `model: "claude-opus-4-6"`)
- Sub-agent `assistant` events carry their own `message.usage` (separate context window)
- The final `result` event's `modelUsage` dict breaks down by model — sub-agents using a different model (e.g. `claude-haiku-4-5`) appear as a separate entry with their own `inputTokens`, `outputTokens`, `costUSD`
- This means we can attribute cost per agent tier without extra work

## Important Technical Notes

### 1. Tool result content measurement from stream-json

The `user` events with `tool_result` blocks contain the actual content returned by tools. The content is either a string or an array of content blocks. To measure size:
```
For each block in content:
  if block is str: chars += len(block)
  if block is dict with "text": chars += len(block["text"])
```
This gives us the actual data volume flowing through each tool call — the key metric for optimization.

### 2. Per-turn `usage` from assistant events

Each `assistant` event includes `message.usage` with a per-turn token snapshot. The `input_tokens` field grows with each turn as context accumulates. By tracking this across turns, we can see exactly where context growth happens — e.g., "Gmail search result on turn 4 added 50K tokens to context."

### 3. Keeping detail logs out of LLM context

`pulse_detail.jsonl` is written to `logs/` which is gitignored. The `run_claude()` function writes the detail record AFTER the Claude invocation completes (after `subprocess.run` returns). The detail data never enters the prompt or conversation. The prompt text is measured before sending, not by including any instrumentation in it.

### 4. Replaces token_usage.jsonl

The existing `token_usage.jsonl` has 2 records with inconsistent schemas — produced by a prior version of the code that no longer exists in `pulse.py`. The new `pulse_detail.jsonl` fully replaces it. The old file can be deleted or ignored.

### 5. Config-controlled verbosity

Per-turn detail (the `turns` array) can be verbose for long conversations. Add a config key `detail_logging: true` (default `true`). When false, only the summary and `tools_summary` are written, not individual turns. This keeps log file size manageable during long collection periods.

### 6. Token estimation is approximate — per-turn usage is authoritative

The `result_est_tokens` (chars/4) is a rough heuristic. It overestimates for JSON boilerplate and underestimates for non-ASCII content. Use it only for relative comparison ("which tool result is biggest"). The authoritative token count comes from the per-turn `input_tokens` delta in `message.usage` — this is what the API actually counted.

### 7. Write failures must not crash the pulse

`_write_detail_record()` must wrap file I/O in try/except, matching the pattern used by `log()` (pulse.py line 121-125). A disk-full or permissions error should log a warning and continue — never block the pulse.

### 8. Sub-agent detection and tracking

When a `tool_use` block has `name == "Agent"`, it spawns a sub-agent. Detection approach:
1. When parsing an `Agent` tool_use, record its `tool_id`, `subagent_type`, and `description` from the input
2. After that, a `system` event with `model: None` signals the sub-agent's init — push onto a stack
3. Subsequent `assistant`/`user` events belong to the sub-agent until its `tool_result` arrives (matched by `tool_use_id`)
4. Pop the stack on the matching `tool_result`

The parent's `result` event `modelUsage` dict already breaks down tokens per model. If the sub-agent uses a different model (e.g. haiku), its cost appears separately. If same model, tokens are combined — in that case we can only attribute by counting sub-agent turns.

The `subagents` array in `PulseDetailRecord` captures: which sub-agents were spawned, their type, model, turn count, and what tools they called. The `model_usage` dict captures per-model cost attribution.

### 9. Update optimization plan with findings

After 2-3 days of data, use `token_report.py` output to update `.docs/plans/2026.03.13-reduce-pulse-token-consumption.md` with corrected root cause analysis. The current optimization plan incorrectly attributes 354K tokens to system/tool schemas — this needs to be revised to reflect that 96% of tokens are conversation content.
