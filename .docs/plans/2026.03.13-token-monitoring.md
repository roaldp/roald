# Token Consumption Monitoring

## High Level Goal

Add visibility into where tokens are being consumed by tracking per-process usage, identifying heavy consumers, and surfacing this data through logs and a CLI report. The system currently has zero token tracking despite spawning many Claude CLI subprocesses.

## Problem Analysis

### Token consumption hotspots (ranked by estimated cost)

1. **`poll_slack_messages` -- CRITICAL**
   Runs every N seconds (config template says 1s, code defaults to 5s -- actual value depends on user's `config.yaml`). Each invocation spawns a full Claude CLI process with prompt + tool definitions just to read 5 Slack messages. At 1s interval: ~3,600 calls/hour. At 5s: ~720 calls/hour. Measured cost per call: ~$0.008 (cache hit). Projected: $5.76-$28.80/hour.

2. **`run_full_pulse` -- HIGH**
   Runs every 30 minutes. Large system prompt (pulse_full.md is ~157 lines) with access to all MCP tools. Scans Gmail, Calendar, Fireflies, Google Drive, Slack. Many tool calls per invocation.

3. **`run_reactive_pulse` -- MEDIUM**
   Triggered per user Slack message. Medium-sized prompt, moderate tool usage. Volume depends on user activity.

4. **`_send_slack_message` -- LOW per call, adds up**
   Each outbound Slack message spawns a Claude CLI process. Used by update notifications, reactive responses, etc. Small prompt but full CLI overhead each time.

### What the system currently tracks

- `run_claude` logs EXEC START/END with wall-clock time and tool start/end counts
- No token counts are captured despite stream-json emitting full usage data in the `result` event
- No aggregation, no per-process breakdown, no alerts

### Phase 0 findings (verified)

Stream-json output (with `--verbose`) emits 4 event types: `system`, `assistant`, `rate_limit_event`, `result`. The `result` event contains:

```json
{
  "type": "result",
  "total_cost_usd": 0.0081575,
  "duration_ms": 2209,
  "usage": {
    "input_tokens": 3,
    "cache_creation_input_tokens": 0,
    "cache_read_input_tokens": 16035,
    "output_tokens": 5
  }
}
```

Key observations:
- `total_cost_usd` is provided directly by the CLI -- no need to compute from token counts
- Cache tokens dominate: 16,035 cache tokens vs 3+5 input/output for a minimal call
- First call creates cache ($0.10), subsequent calls read cache ($0.008) -- 12x cheaper
- The existing `run_claude` already uses `--verbose` (line 211) and already has an `elif event_type == "result"` branch (line 283) -- minimal code change needed

## Important Technical Notes

### Verified stream-json field paths

The `result` event fields we extract (confirmed via Phase 0 testing):

| Field | Path | Description |
|-------|------|-------------|
| `input_tokens` | `event["usage"]["input_tokens"]` | Prompt tokens (excluding cache) |
| `output_tokens` | `event["usage"]["output_tokens"]` | Response tokens |
| `cache_creation_input_tokens` | `event["usage"]["cache_creation_input_tokens"]` | Tokens used to create prompt cache |
| `cache_read_input_tokens` | `event["usage"]["cache_read_input_tokens"]` | Tokens read from prompt cache |
| `cost_usd` | `event["total_cost_usd"]` | Per-invocation dollar cost (computed by CLI) |
| `duration_ms` | `event["duration_ms"]` | Wall clock time |

If the `usage` field is absent, fall back to all-zero `TokenUsage` and log a warning. If cost is zero for 5+ consecutive invocations, log a persistent warning via a module-level counter (`_zero_cost_streak: int`). Counter resets to 0 on any non-zero cost.

### Elapsed time vs duration_ms

The JSONL `elapsed_s` field uses wall-clock time from `time.monotonic()` (already computed in `run_claude`), not `duration_ms` from the result event. Wall-clock includes subprocess spawn overhead, which is relevant for understanding actual system cost. The CLI's `duration_ms` measures only API time.

### Operation name limitation

`_send_slack_message` hardcodes `operation="update_slack_message"` for all Slack sends. The token report will show all sends under one bucket. This is a pre-existing limitation -- not addressed in this plan.

### JSONL format

Each line in `logs/token_usage.jsonl`:
```json
{"timestamp": "2026-03-13T14:22:01", "operation": "poll_slack_messages[D_CHAN]", "input_tokens": 1200, "output_tokens": 80, "cache_creation_input_tokens": 0, "cache_read_input_tokens": 16035, "cost_usd": 0.008, "elapsed_s": 3.2, "tool_starts": 1}
```

Note: field names match the Claude API naming for readability (per coding conventions: "readability over cleverness").

### Log rotation

At 720+ entries/hour, the JSONL file can grow to ~50-100 MB/day. Follow-up concern, not part of initial implementation.

### Concurrency

`pulse.py` uses asyncio (single-threaded event loop) and `run_claude` calls blocking `subprocess.run`. No concurrent access issues for the in-memory counter. No locking needed.

## Success Criteria

1. Every `run_claude` invocation logs its token usage and cost to `logs/token_usage.jsonl`
2. `python3 scripts/token_report.py --since 24h` produces a readable breakdown showing which operations cost the most
3. The EXEC END log line includes cost and token counts for each invocation
4. Running the system for 1 hour and checking the report reveals the actual dominant consumer
5. No behavioral changes to the existing system -- this is observability only
